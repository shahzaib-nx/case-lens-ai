# Engineering Notes & Bug History

This document serves as the historical technical log for CaseLens AI. It tracks significant architectural hurdles, critical bugs, and their implemented solutions.

---

## 1. Vercel Build Crash: Top-Level OpenAI Instantiation

**Issue:**
The application successfully compiled locally but crashed consistently on Vercel during the "Collecting page data" optimization step. The build output threw a `Missing credentials. Please pass an apiKey` error originating from `lib/ai.ts` and `app/api/generate-overall-analysis/route.ts`.

**Root Cause:**
The `OpenAI` (Groq) client was being instantiated at the top level of the module (e.g., `const ai = new OpenAI({...})`). During Vercel's static page analysis phase, Next.js evaluates all top-level module code. Because Vercel's build environment did not have the `GROQ_API_KEY` environment variable explicitly exposed to the static builder, the OpenAI SDK immediately threw a fatal missing credentials error.

**Solution:**
Refactored the codebase to strictly enforce **lazy instantiation**. The `OpenAI` client is now instantiated dynamically inside a getter function (`getAiClient()`) or directly inside the active `POST` request handler. This ensures the client is only initialized at runtime when a real HTTP request is actively being processed, bypassing the static build step constraints completely.

## 2. Legacy ID Mapping Failures (The "q-idx" Bug)

**Issue:**
Older Multiple Choice Questions (MCQs) generated by early AI models lacked explicit `id` UUID fields. When the user submitted a practice attempt for these older cases, the Results page failed to map the user's attempt payload back to the original question data, causing the UI to display blank reviews or crash.

**Root Cause:**
The Zustand store's `submitPracticeAttempt` action relied strictly on `question.id`.

**Solution:**
Upgraded the backend and the Zustand store to implement a backward-compatible fallback ID generator. When finding questions, the codebase now maps against `q.id || q-${idx}`. This ensures legacy questions generated without IDs seamlessly map to the index-based IDs created safely on the practice page.

## 3. The "Ghost Session" Analytics Bug

**Issue:**
When a user completed a quiz and navigated rapidly from the Practice Page to the Results Page, the Results page would display "0 answered" despite the database properly saving the attempts.

**Root Cause:**
The React component lifecycle unmounted abruptly during navigation, inadvertently triggering the auto-generation of an empty "Ghost Session" in the Zustand store. The Results page relied on a flawed `getLatestSession` utility that used heuristic date-sorting. Because the "Ghost Session" was created milliseconds *after* the actual completed session, the heuristic algorithm grabbed the empty session and rendered it instead.

**Solution:**
Deprecated purely date-based heuristic lookups. Upgraded the Next.js router to explicitly pass the exact `sessionId` query parameter directly in the URL (`/case/results?id=...&sessionId=...`). The Results page now deterministically reads the exact session provided by the URL, completely bypassing any race conditions.

## 4. TypeScript Strict Compilation Crashes

**Issue:**
Next.js Turbopack silently failed to compile, hiding updates from the UI and blocking deployments.

**Root Cause:**
Duplicate helper functions and union-type mismatches (`McqQuestion | LegacyQuestion`) were violating strict type bindings. Furthermore, several components were trying to cast objects as interfaces (like `QuestionReviewAnalysis`) without properly importing them from the store.

**Solution:**
Executed a deep audit using `npx tsc --noEmit` across the entire codebase to expose and eradicate hidden strict-typing errors, enforce explicit imports, and ensure all interface properties aligned safely.

---

## 5. Monolithic Storage Bottlenecks

**Issue:**
Deeply nested array mutation algorithms were causing significant UI jank (stutter) and threatening imminent browser `QuotaExceededError` crashes.

**Root Cause:**
Zustand was storing the entire global application state (Cases, Sessions, Attempts, Questions, and massive Markdown Strings) inside a single 5MB `localStorage` JSON blob. Every time a user clicked "Submit Answer", the application was forced to `O(N)` map over the entire history of the application just to serialize and save one tiny object update.

**Solution Plan (Pending):**
Designed the Phase-2 IndexedDB Normalization Plan. The architecture will transition away from monolithic JSON blobs towards asynchronous, relational `O(1)` Object Stores (e.g., splitting `cases`, `mcqs`, and `attempts` into distinct tables), reserving Zustand strictly for ephemeral UI state like loading spinners.
